The system implements a four-stage pipeline for deploying, evaluating, and monitoring an AI agent on Google Cloud. The pipeline flows left to right through four distinct stages, each feeding its output into the next.

In Stage 1, the developer defines a FinanceAgent class with two core methods: set_up, which initializes a Gemini 2.5 Flash generative model and configures OpenTelemetry environment variables, and query, which accepts a user prompt and returns the model's response. The FinanceAgent is packaged and deployed to Vertex AI Agent Engine via the ReasoningEngine.create API, which uploads the agent code along with its Python dependencies to a managed runtime. The deployment returns a resource name that serves as the agent's remote endpoint.

In Stage 2, the pipeline generates traffic by sending test prompts to the deployed agent through the engine.query interface. Two representative queries are issued: one about billing account status and one about opening a support ticket. The agent's responses are collected alongside the original prompts and ground-truth reference answers, then assembled into a pandas DataFrame with three columns: prompt, response, and reference.

In Stage 3, this DataFrame flows into the Vertex AI EvalTask evaluation engine. The EvalTask applies two evaluation methods in parallel. The first is a custom PointwiseMetric rubric that scores each response on a scale of 1 to 5 across two criteria: helpfulness, measuring whether the response directly and accurately answers the request, and conciseness, measuring whether the response is brief without unnecessary rambling. A Model-as-a-Judge LLM reads each prompt-response pair and assigns a numeric score with a written explanation. The second evaluation method is an exact_match baseline that performs algorithmic string comparison against the reference answers.

In Stage 4, the scored results DataFrame is enriched with an eval_timestamp and agent_version column, then exported to BigQuery via the pandas-gbq library using an append operation. The data lands in the agent_metrics.eval_rubric_results table, where Looker connects to visualize quality trends over time, grouped by agent version and release date.

Running orthogonally across all stages, OpenTelemetry tracing captures operational spans for every agent invocation, reasoning loop iteration, LLM generation call, and tool execution. These spans flow automatically to Google Cloud Trace for latency analysis and bottleneck identification.
